# وثيقة المواصفات الفنية وخطة العمل: نظام إدارة الورشة (Workshop ERP)

## 📌 1. نظرة عامة على النظام (Project Overview)
نظام أتمتة تشغيلي وإداري متكامل لورش صيانة السيارات، يعتمد على تقنيات الباركود (QR/Barcode) والرسائل النصية التلقائية. يهدف النظام إلى تنظيم التدفق التشغيلي، الفصل الصارم بين الصلاحيات (الاستقبال، الفنيين، الإدارة)، وتوفير شاشات تتبع ودفع خارجي آمنة للعملاء.

---

## 🔄 2. دورة حياة المعاملة (User Journey & State Machine)

ينتقل "أمر الشغل" (Job Card) داخل النظام عبر 6 حالات برمجية محكمة لمنع التداخل أو الارتجال:

1. **مُسجل (Registered):** * يستقبل موظف الكاونتر العميل، يسجل بياناته وبيانات المركبة (أو يستدعيها إن كانت مسجلة مسبقاً).
   * يفتح أمر شغل جديد ويطبع للعميل كرت يحتوي على (البيانات الأساسية + Barcode فريد).
2. **تحت الفحص (In_Inspection):**
   * يستلم الفني الكرت، ويمسح الباركود عبر جواله/التابلت لتفتح له صفحة الفحص فوراً.
   * يقوم الفني بتصوير المركبة من جهاتها الأربع (لحماية الورشة قانونياً من أي دعاوى سابقة).
   * يحدد الفحص المبدئي، القطع التقريبية، والوقت المتوقع للتسليم.
3. **جاري العمل (In_Progress):**
   * يبدأ العمل الفعلي على المركبة.
   * يرسل النظام تلقائياً رسالة نصية (SMS) للعميل تحتوي على: (الوقت المتوقع، تفاصيل الصيانة، ورابط مباشر لتتبع الحالة).
4. **جاهز للاستلام (Ready_For_Pickup):**
   * ينهي الفني الصيانة، يغلق الملف، ويحدد التسعيرة النهائية والقطع الفعلية التي استُخدمت.
   * تصل للعميل رسالة نصية تفيد بنجاح الصيانة، مع ملخص مالي، ورابط للدفع الإلكتروني (Online).
5. **تم الدفع (Paid):**
   * تحدد حالة الدفع تلقائياً كـ "مدفوع أونلاين" إذا سدد العميل عبر الرابط.
   * في حال حضور العميل والدفع نقداً أو عبر الشبكة في الفرع، يقوم موظف الاستقبال بتحديث الحالة يدوياً إلى "مدفوع بالفرع" ويطبع كرت الخروج.
6. **مغلق ومستلم (Delivered & Closed):**
   * يقدم العميل كرت الخروج للفني.
   * يمسح الفني الباركود لتأكيد خروج المركبة وتسليم المفتاح، ويغلق الملف نهائياً، وتصل للعميل رسالة شكر تلقائية.

## 🗄️ 3. هندسة قاعدة البيانات (Database Schema)

> 🛠️ **تصحيح هندسي:** تم إلغاء الحقول النصية المتعددة للقطع في جدول أمر الشغل، واستبدالها بجداول وسيطة (Junction Tables) لضمان دقة جرد المستودع، وحساب الضرائب، وعدم تضارب البيانات عند تغير الأسعار مستقبلاً.

### أولاً: البيانات الأساسية (Master Data)

#### 1. جدول العملاء (Customers)
* `id`: Primary Key (Int / UUID)
* `name`: String (اسم العميل)
* `phone`: String (رقم الجوال - Unique)
* `email`: String (البريد الإلكتروني)
* `rank`: Enum (تصنيف العميل: برونزي، فضي، ذهبي - يرتفع تلقائياً بكثرة الزيارات لتقديم خصومات)

#### 2. جدول المركبات (Vehicles)
* `id`: Primary Key
* `customer_id`: Foreign Key (يربط المركبة بجدول العملاء)
* `brand`: String (الماركة - مثل: تويوتا)
* `model`: String (الموديل - مثل: كامري)
* `year_created`: Int (سنة الصنع)
* `plate_number`: String (رقم اللوحة)

#### 3. جدول المستودع والقطع (Parts_Inventory)
* `id`: Primary Key
* `name`: String (اسم القطعة)
* `stock_qty`: Int (الكمية المتوفرة بالمخزن)
* `base_price`: Decimal (سعر التكلفة)
* `compatible_cars`: String/Array (السيارات المتوافقة مع القطعة)
* `image_url`: String (رابط الصورة المصغرة للقطعة)

#### 4. جدول قائمة الخدمات (Services_List)
* `id`: Primary Key
* `name`: String (اسم الخدمة - مثل: تغيير زيت، ميزان)
* `description`: Text (وصف الخدمة)
* `required_mechanics`: Int (عدد الفنيين المطلوبين للخدمة)
* `price`: Decimal (سعر الخدمة الثابت / أجور اليد)

---

### ثانياً: البيانات التشغيلية وحركات النظام (Operational Data)

#### 5. جدول بطاقات العمل / أوامر الشغل (Job_Cards)
* `id`: Primary Key (يُشتق منه الباركود)
* `vehicle_id`: Foreign Key (يشير لجدول المركبات)
* `technician_id`: Foreign Key (يشير لجدول الموظفين/الفنيين)
* `status`: Enum (Registered, In_Inspection, In_Progress, Ready, Paid, Closed)
* `expected_delivery`: DateTime (الوقت المتوقع للتسليم)
* `total_price`: Decimal (المبلغ الإجمالي المحسوب برمجياً من الخدمات والقطع)

#### 6. جدول تفاصيل قطع بطاقة العمل (Job_Card_Parts) [جدول وسيط]
* `id`: Primary Key
* `job_card_id`: Foreign Key (يشير لأمر الشغل الحالي)
* `part_id`: Foreign Key (يشير للقطعة من المستودع)
* `quantity`: Int (الكمية المستهلكة في هذه الزيارة)
* `sold_price`: Decimal (سعر البيع وقت العملية - لحماية الفواتير القديمة من تغير أسعار السوق)

#### 7. جدول تفاصيل خدمات بطاقة العمل (Job_Card_Services) [جدول وسيط]
* `id`: Primary Key
* `job_card_id`: Foreign Key
* `service_id`: Foreign Key

#### 8. جدول صور الفحص (Inspection_Photos)
* `id`: Primary Key
* `job_card_id`: Foreign Key (مربوط بأمر الشغل لفرز صور كل زيارة على حدة)
* `photo_url`: String (رابط الصورة على السيرفر السحابي)
* `uploaded_by`: Foreign Key (معرف الفني الذي التقط الصورة)
* `created_at`: DateTime

#### 9. جدول سجل العمليات (System_Logs)
* `id`: Primary Key
* `user_id`: Foreign Key (الموظف الذي قام بالحركة)
* `action`: String (وصف الحركة - مثل: "قام أحمد بتغيير الحالة إلى جاري العمل")
* `created_at`: DateTime

## 🌐 4. بنية النظام والأمان (System Architecture)

> 🛠️ **تصحيح هندسي:** لجعل العميل قادراً على الدفع وتتبع سيارته من خارج الورشة، لا يمكن جعل النظام محلياً (Local Network) بالكامل. الحل هو نظام هجين (Hybrid Security):

* **الاستضافة:** سحابية بالكامل (Cloud Hosting) على سيرفر خاص VPS أو منصة Vercel.
* **الأمان الجغرافي (Middleware IP Filtering):**
  * يتم برمجة **Middleware في Next.js** ليفحص الـ IP الخاص بكل طلب.
  * لوحات تحكم الإدارة، الاستقبال، وشاشات الفنيين **تمنع منعاً باتاً** أي اتصال قادم من خارج شبكة إنترنت الورشة الثابتة (Static IP).
  * روابط العملاء (صفحة الفاتورة، تتبع السيارة، بوابة الدفع الإلكتروني) تُستثنى من الحظر وتكون متاحة للعامة من أي مكان.

## 🛠️ 5. حزمة التقنيات الموصى بها (Tech Stack)

* **إطار العمل الأساسي:** `Next.js` (Full-Stack للربط المباشر لصفحات العرض بالخلفية وقاعدة البيانات في مشروع واحد).
* **الـ ORM وقاعدة البيانات:** `Prisma` أو `Drizzle ORM` مع قاعدة بيانات `PostgreSQL` أو `MySQL` لسهولة بناء العلاقات المعقدة بين الجداول.
* **التصميم والواجهات:** `Tailwind CSS` مدمج مع مكتبة المكونات الاحترافية الجاهزة `Shadcn/ui` لسرعة بناء الجداول، القوائم المنبثقة، ولوحات التحكم دون تضييع الوقت في التصميم من الصفر.
* **إدارة الصلاحيات:** `NextAuth.js` للتحقق من هوية المستخدمين وتوزيع الأدوار (Admin, Receptionist, Technician).
* **مزامنة البيانات الآنية:** `React Query (TanStack Query)` لتحديث شاشات الاستقبال تلقائياً فور قيام الفني بأي حركة، دون الحاجة لعمل التحديث اليدوي (Refresh).
* **التقارير المجدولة:** `Cron Jobs` مبرمجة لقراءة جدول الـ `System_Logs` كل نهاية أسبوع، وتلخيصه، وإرساله آلياً كتقرير مختصر لإيميل المدير.

## 📅 6. خطة التنفيذ خطوة بخطوة (Roadmap)

* **المرحلة 1 [الأساسات]:** إعداد بيئة مشروع Next.js، كتابة الـ Schemas في الـ ORM، وعمل الـ Migrations لإنشاء الجداول.
* **المرحلة 2 [الأمان]:** إعداد نظام الصلاحيات بالـ NextAuth وكتابة الـ Middleware لعزل الـ IP الخاص بالورشة عن لوحات التحكم.
* **المرحلة 3 [شاشات الاستقبال]:** بناء واجهات تسجيل العميل والسيارة، وإنشاء أمر الشغل (Job Card) وتوليد الباركود.
* **المرحلة 4 [شاشات الفني]:** بناء نظام رفع الصور للفحص، وقوائم اختيار القطع والخدمات، وزر تحديث الحالات.
* **المرحلة 5 [بوابة العميل]:** بناء الصفحة الخارجية العامة للفاتورة، والربط التجريبي مع بوابة دفع ورسائل SMS.
* **المرحلة 6 [الإدارة والتقارير]:** بناء لوحة تحكم المدير، جدول الـ Logs، وأتمتة تقرير الأسبوع.
